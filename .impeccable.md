# MeldUI Design Context

## Design Context

### Users
Solo developers and small dev teams building software with AI-assisted workflows. They're technical, opinionated about their tools, and expect professional-grade UX. They're using MeldUI as their daily driver for issue tracking and AI-powered development — it needs to earn trust through competence, not flashiness.

### Brand Personality
**Purposeful. Capable. Unobtrusive.**

MeldUI should feel like a precision instrument — powerful but never overwhelming. It should communicate technical confidence: users trust the platform because it feels considered and reliable. The interface is welcoming and approachable (not a terminal), but never dumbed down. Every element earns its place.

**Emotional goal:** Trust through competence. Users should feel like the tool is working *with* them, not demanding attention.

### Aesthetic Direction

**Visual tone:** Modern developer tool with warmth — sits between Linear's polish and Conductor's dark sophistication. Information-dense but not cramped.

**References:**
- **Linear** — Clean hierarchy, purposeful use of color for status, fast and confident transitions, tight spacing that still breathes
- **Conductor** — Dark-mode sophistication, developer-centric identity, backdrop blur effects, contained card aesthetic with subtle borders

**Anti-references:**
- Terminal UIs / CLI aesthetics — MeldUI is a GUI that should feel welcoming
- Claude Code's raw terminal interface — power should come through design, not plaintext
- Overly playful or consumer-app aesthetics (bouncy animations, emoji-heavy, rounded-everything)

**Theme:** Dark mode is the primary experience. Light mode supported but dark is the hero. The current neutral grayscale palette is a solid foundation — layer in more color personality through:
- Richer status/phase colors with more saturation
- Subtle gradient treatments on key surfaces
- The emerald brand accent used more boldly as a signature element
- Secondary accent colors for AI/agent activity to distinguish human vs machine actions

**Density:** Information-dense. Lean compact — respect screen real estate. Tight line heights, smaller default text where appropriate, more data visible per viewport. Think Linear's density, not Notion's spaciousness.

### Design Principles

1. **Density over decoration** — Show more information, use fewer pixels. Every padding value and margin should justify its space. Prefer compact, scannable layouts that let power users move fast.

2. **Color with intent** — Color is a functional tool, not decoration. Use it to encode meaning (status, type, priority, AI vs human), not to fill space. The emerald accent is the brand signature — use it sparingly so it carries weight.

3. **Confidence through consistency** — Uniform spacing, predictable component behavior, and systematic color application build trust faster than any individual design choice. When users see a pattern once, they should be able to predict it everywhere.

4. **Welcoming power** — Expose capability without overwhelming. Progressive disclosure over hidden features. The interface should feel approachable on first use but reveal depth as users explore. Never sacrifice power for simplicity — find designs that deliver both.

5. **Quiet until relevant** — Animations, transitions, and visual effects should be purposeful and brief. No gratuitous motion. Loading states, status changes, and AI activity indicators should communicate state, not entertain. The interface gets out of the way so the user can focus on their work.

### Color System

**Brand:** Emerald (#10b981 / #5BF0A0) — the signature. Use for primary actions, success states, and brand moments.

**Semantic palette (functional, not decorative):**
- Emerald — success, completion, brand
- Blue — in-progress, active work, information
- Purple — research, planning, AI thinking
- Amber — warnings, deferred, needs attention
- Red — errors, destructive actions, blockers
- Cyan/Teal — secondary information, metadata

**Surface treatment (dark mode):**
- Use subtle elevation through border opacity and background lightness shifts (not shadows)
- Cards and panels differentiate through 1-2 OKLCH lightness steps
- Borders at low opacity (10-15% white) for definition without harshness

### Typography

**Geist Variable** — clean, modern, excellent for dense UI. Use weight and size for hierarchy, not typeface changes.

- Monospace (Geist Mono or system) only for code, file paths, and technical identifiers
- Uppercase tracking-wider only for section labels — use sparingly
- Prefer font-medium over font-bold for emphasis in compact layouts

### Spacing & Density Guidelines

- Default gap: 1-2 (4-8px) between related items
- Section padding: 3-4 (12-16px)
- Card padding: 2-3 (8-12px) for compact cards
- Sidebar items: tight vertical rhythm, ~28-32px row height
- Target: Linear-level density — more information per viewport than typical apps
